# Patient & Doctor Health OS — РФ-версия

**Автор:** Богдан
**Дата:** 24 апреля 2026
**Статус:** план MVP, двусторонний (пациент + врач), self-hosted, open-source
**Рынок:** Россия (Москва/МО как базовый кейс, остальные регионы — по мере доступности данных)

## 0. Что меняется по сравнению с общей версией

РФ-ландшафт отличается от UAE принципиально в трёх вещах:

1. **Доступ пациента к своим данным реально есть.** ЕМИАС в Москве отдаёт всю электронную медкарту через [mos.ru](https://www.mos.ru/city/projects/medcarta/) — осмотры, анализы, снимки (включая DICOM), справки с ЭЦП. В других регионах — через [Госуслуги](https://www.gosuslugi.ru) (РЭМД/ИЭМК), объём меньше, но растёт. Это делает ingest **реальным**, а не воображаемым.
2. **Регуляторика.** [152-ФЗ](http://www.consultant.ru/document/cons_doc_LAW_61801/) (персданные, спецкатегория — здоровье), [323-ФЗ](http://www.consultant.ru/document/cons_doc_LAW_121895/) ст. 13 (врачебная тайна), Приказ Минздрава №947н (ЭМК). Для личного self-hosted использования — ты обрабатываешь **свои** данные, 152-ФЗ не применяется в части обработки оператором. Как только появляется второй пациент (родственник, врач ведёт своих) — нужен оператор, уведомление Роскомнадзору и письменное согласие в спецформе.
3. **Лабы шлют результаты на email и в личные кабинеты.** [Инвитро](https://www.invitro.ru), [Гемотест](https://www.gemotest.ru), [KDL](https://kdl.ru), [Хеликс](https://helix.ru), [CMD](https://www.cmd-online.ru) — у всех PDF на email + XML/JSON в ЛК. Это самый дешёвый ingest-канал.

Главный трейдофф тот же: **Notion не источник истины.** Хранить медданные в Notion нельзя (серверы в США, 152-ФЗ это прямо не разрешает для спецкатегории). Notion — только витрина с агрегатами.

### Почему FHIR, а не «свои таблицы»

ЕГИСЗ/РЭМД внутри говорит на [СЭМД](https://portal.egisz.rosminzdrav.ru/materials/3307) (структурированный электронный медицинский документ, это профиль CDA/HL7). Это не FHIR, но конвертируется. HAPI FHIR как локальное ядро даёт:
- готовую модель (Observation, Condition, MedicationStatement и т.д.);
- возможность импортировать `bundle` из Apple Health и Google Fit без переписывания;
- когда ЕГИСЗ откроет полноценный пациентский API (это обсуждается), маппинг уже будет.

Альтернатива — простой Postgres со своей схемой. Быстрее на 2–3 дня, но через 6 месяцев будешь переписывать. Не стоит.

### Стек (всё open-source, всё локально, всё работает без VPN)

| Слой | Решение | Почему |
|---|---|---|
| Оркестратор | Python + cron (или [Prefect](https://prefect.io) при >5 агентов) | На старте хватит cron |
| Domain store | [HAPI FHIR](https://hapifhir.io) + Postgres 16, Docker | Стандарт |
| Шифрование | LUKS/VeraCrypt на диске + pgcrypto для ФИО/СНИЛС/полиса | Диск + чувствительные поля |
| Ingest PDF | [Docling](https://github.com/docling-project/docling) + [PaddleOCR](https://github.com/PaddlePaddle/PaddleOCR) | Оба хорошо справляются с русским |
| Нормализация лаб | Regex + LLM → маппинг на [LOINC](https://loinc.org) + рос. справочник НСИ | У РФ есть [Росмедлиб](https://nsi.rosminzdrav.ru) — справочник медизделий и показателей |
| LLM | [Ollama](https://ollama.com) → [Qwen2.5 14B](https://ollama.com/library/qwen2.5) или [Saiga-Llama3](https://huggingface.co/IlyaGusev/saiga_llama3_8b) + [MedGemma 4B](https://huggingface.co/google/medgemma-4b-it) | Qwen/Saiga — русский без танцев, MedGemma — мед.контекст |
| Speech | [whisper.cpp](https://github.com/ggerganov/whisper.cpp) large-v3 | Хорошо понимает русскую мед.терминологию |
| Email ingest | [imap-tools](https://github.com/ikvk/imap_tools) + парсер вложений | Лабы шлют на email |
| ЕМИАС pull | Python-скрипт с авторизацией через mos.ru API (или Selenium при необходимости) | Нет публичного API, но есть обход через ЛК |
| Госуслуги pull | Ручной экспорт PDF → папка → ingest (публичного API для пациента нет) | Самое частое ограничение РФ-ландшафта |
| Wearables | [Health Auto Export](https://www.healthexportapp.com) (iOS) / [Health Sync](https://play.google.com/store/apps/details?id=nl.appyhapps.healthsync) (Android) → webhook | Работают без Apple ID US |
| Notion sync | Официальный [Notion API](https://developers.notion.com) | Только проекции |
| Календарь | Google Calendar или [Яндекс.Календарь CalDAV](https://yandex.ru/support/calendar/) | GCal проще, CalDAV если Google недоступен |
| Алерты | Telegram Bot | У тебя уже в стеке |
| Секреты | [SOPS](https://github.com/getsops/sops) + age | Локальные ключи |
| Деплой | Docker Compose на домашнем Mac mini / NUC / старом ноуте | Сервер включён 24/7, не ноут |

## 2. Источники данных — реалистичная картина РФ

### 2.1 ЕМИАС (Москва и МО) — главный источник, если ты в Москве

Через [ЕМК на mos.ru](https://www.mos.ru/city/projects/medcarta/) доступно:
- Осмотры врачей (протоколы с ЭЦП).
- Все анализы из поликлиник ОМС.
- Результаты инструментальных исследований + **архив снимков DICOM** по отдельному запросу (ссылка живёт 5 дней).
- Выписки из стационара.
- Справки, больничные.
- Записи на приём (предстоящие и прошлые).
- Временный доступ врачу по сгенерированной ссылке (это уже почти то, что ты строишь, но без AI-слоя).

**Как тянуть:** публичного пациентского API нет. Варианты:
1. **Браузерный флоу через Playwright** — логинимся через Госуслуги/mos.ru, забираем PDF/JSON из кабинета. Надёжно, но хрупко при редизайне.
2. **Мобильное приложение «ЕМИАС.ИНФО»** — у него есть внутренний API, его можно реверсить (HTTP через mitmproxy). Легально для личных данных, но публикация такого в open-source — серая зона.
3. **Только ручной экспорт** — раз в месяц скачиваем ZIP с медкартой, кидаем в папку `/inbox/emias/`. Скучно, но работает с Дня 1.

**Рекомендация для MVP:** старт — вариант 3. Вариант 1 — на 2-м месяце, когда стабилизируется ядро.

### 2.2 Госуслуги / ЕГИСЗ (регионы) — ограниченно

Через [Госуслуги](https://www.gosuslugi.ru) в разделе «Моё здоровье» доступны документы из РЭМД (Реестр электронных медицинских документов). Объём зависит от региона и подключённости МО к ЕГИСЗ. Публичного API для пациента нет. Вариант — ручной экспорт PDF.

### 2.3 Частные лаборатории — самый простой и стабильный канал

| Лаб | Канал получения | Формат | Автоматизация |
|---|---|---|---|
| [Инвитро](https://www.invitro.ru) | Email + ЛК | PDF + XML (ИНИС для клиник) | Email-парсер |
| [Гемотест](https://www.gemotest.ru) | Email + [Telegram-бот](https://t.me/gemotest_bot) + ЛК | PDF | Telegram forward или email |
| [KDL](https://kdl.ru) | Email + ЛК | PDF | Email-парсер |
| [Хеликс](https://helix.ru) | Email + ЛК | PDF | Email-парсер |
| [CMD](https://www.cmd-online.ru) | Email + ЛК | PDF | Email-парсер |
| [Citilab](https://citilab.ru) | Email + ЛК | PDF | Email-парсер |

**План:** завести отдельный ящик `labs@<домен>` (или алиас), при заказе указывать его. `ingest.labs` тянет IMAP раз в 15 минут, парсит вложения.

### 2.4 Частные клиники и телемед

- [СберЗдоровье](https://sberzdorovie.ru) (бывш. DocDoc) — заключения в PDF в ЛК и на email.
- [Яндекс.Здоровье](https://health.yandex.ru) — аналогично.
- [Доктис](https://doctis.ru), [Онлайн Доктор](https://ondoc.me) — PDF на email.
- [СМ-Клиника](https://www.smclinic.ru), [ЕМС](https://www.emcmos.ru), [К+31](https://klinika.k31.ru), [Медси](https://medsi.ru) — у каждого свой ЛК, PDF на email или скачивание. Медси и ЕМС пилили интеграцию с ЕГИСЗ — часть документов уходит в РЭМД автоматически.

Для всех — единый канал **email → ingest** остаётся самым стабильным.

### 2.5 Wearables

- **Apple Watch / iPhone:** [Health Auto Export](https://www.healthexportapp.com) → webhook на локальный URL (через [Cloudflared tunnel](https://github.com/cloudflare/cloudflared) или локальную сеть). Работает из РФ без VPN.
- **Android:** [Health Sync](https://play.google.com/store/apps/details?id=nl.appyhapps.healthsync) из Google Fit/Samsung Health в Google Sheets/webhook.
- **Xiaomi Mi Band / Amazfit:** [Gadgetbridge](https://gadgetbridge.org) — open-source Android-клиент, не требует облаков Xiaomi.
- **Whoop / Oura / Garmin** — API есть, но нужен аккаунт с европейскими реквизитами. Работают, если у тебя он есть.

### 2.6 Фарма и рецепты

Единой базы рецептов пациента в открытом доступе нет. Варианты:
- Фото упаковки/рецепта → OCR → `MedicationStatement`.
- [Apteka.ru](https://apteka.ru) / [Еаптека](https://www.eapteka.ru) — историю заказов можно выдирать из ЛК, но не стоит того на MVP.

## 3. Модель данных (FHIR + адаптация под РФ)

Стандартные FHIR-ресурсы + расширения:

| FHIR ресурс | Что внутри | Notion-проекция | РФ-специфика |
|---|---|---|---|
| `Patient` | ФИО, DOB, пол, контакты | «Профиль» | СНИЛС, полис ОМС, серия/номер паспорта — в зашифрованных полях |
| `Practitioner` | ФИО, специальность, МО | «Врачи» | ФРМР ID если есть |
| `Organization` | Клиника/лаб | «Клиники» | ОГРН, адрес, лицензия |
| `Encounter` | Визит: дата, МО, врач, повод | «Визиты» | Оплата: ОМС/ДМС/платно |
| `Observation` | Показатель + значение + норма | «Метрики» | LOINC + рус. название (из [НСИ Минздрава](https://nsi.rosminzdrav.ru)) |
| `DiagnosticReport` | Лаб/инстр. отчёт | «Анализы» | Лаб код, тип исследования |
| `Condition` | Диагноз | «Диагнозы» | МКБ-10 (Россия пока не перешла на МКБ-11) |
| `MedicationStatement` | Приём препарата | «Лекарства» | МНН + торговое название (из [ГРЛС](https://grls.rosminzdrav.ru)) |
| `ImagingStudy` | Снимки (МРТ/КТ/рентген) | «Исследования» | Ссылка на DICOM в MinIO |
| `CarePlan` | План лечения | «План» | Назначения с датами |
| `Flag` (кастом) | Алерт | «Алерты» | Триггер, серьёзность |

### Что не кладём в Notion никогда

- Паспорт, СНИЛС, полный номер полиса.
- Сырые PDF заключений (только ссылка на локальный URL, доступный из LAN).
- DICOM.
- Полные транскрипты диктовок (только суммаризация).
- Диагнозы с кодами, которые по [323-ФЗ](http://www.consultant.ru/document/cons_doc_LAW_121895/) подпадают под усиленную защиту (ВИЧ, психиатрия, наркология) — если такие возможны в кейсе, для них отдельный шифр-режим либо вывод их из Notion полностью.

## 4. Агенты Computer

### 4.1 `ingest.labs.email` — главный агент с Дня 1

**Триггер:** cron каждые 15 минут, IMAP-пулл с `labs@домен`.

```
Goal: PDF из письма лаборатории → структурированный DiagnosticReport
Success: в FHIR лежит отчёт с ≥1 Observation, PDF в MinIO, флаги подняты
Plan:
1. IMAP: забрать новые письма, определить отправителя (Инвитро/Гемотест/KDL/...)
2. Достать PDF-вложения (у Инвитро также может быть XML — парсим его напрямую)
3. Docling → plain text + таблицы (для русских PDF работает)
4. LLM extract: {показатель_рус, значение, единица, референс_мин, референс_макс}
5. Маппинг на LOINC через словарь (20 популярных хардкодом) + fallback через LLM
6. Сравнение с референсом → Flag
7. Запись DiagnosticReport + Observations в HAPI
8. Синк Notion-проекций
Verify: письмо от Инвитро с биохимией → через 2 минуты в «Анализах» новая строка
Время: 3 дня на v1
```

**Риск:** формат PDF у лаб регулярно меняется. Решение — словарь-экстрактор для каждой лаборатории отдельно, с фоллбеком на чистую LLM-экстракцию.

### 4.2 `ingest.emias.manual` — ручной экспорт с mos.ru

**Триггер:** файл попал в `/inbox/emias/`.

Разбираем экспортный ZIP, парсим содержащиеся документы (у них более-менее стандартизованный формат CDA-like). Маппим в FHIR. На v2 — автоматизируем через Playwright.

### 4.3 `ingest.wearables` — как в общей версии

Webhook от Health Auto Export → дневные агрегаты → Observation.

### 4.4 `agent.visit_prep` — подготовка к визиту

```
Goal: за 24ч до события в календаре с тегом #visit — бриф
Success: страница в Notion, пуш в Telegram обеим сторонам
Plan:
1. GCal pull → события с #visit в ближайшие 24ч
2. Сбор из FHIR: последние лабы (90 дней), динамика ключевых Observation, открытые Conditions, текущие MedicationStatement
3. LLM (на русском): 5–7 вопросов врачу из заметок пациента + трендов
4. LLM: 1-страничный SOAP-срез для врача
5. Notion: пациент видит свою часть (жалобы, вопросы, лекарства), врач — клиническую
6. Telegram пуш
Verify: за сутки до записи к эндокринологу — бриф в Notion
Время: 2 дня
```

### 4.5 `agent.alerts` — алерты по отклонениям

Правила v1:
- Observation вне референса → Flag (severity по дельте в сигмах от нормы).
- Тренд: 3 подряд ухудшения ключевой метрики → Flag medium.
- Критические пороги: глюкоза >11 ммоль/л, SpO2 <92%, ЧСС покоя >130 → Flag high + немедленный TG.
- Пропуск приёма лекарства (нет подтверждения за 6ч от назначенного) → Flag low.

Формулировка алертов: **«Показатель X = Y, вне нормы [A–B]. Обсудите с врачом.»** Никогда не ставим диагноз.

### 4.6 `agent.visit_post` — постобработка визита

Диктовка или текст после приёма → whisper.cpp → LLM извлекает SOAP, изменения лекарств, назначенные исследования, следующий визит. **Human-in-the-loop:** любые изменения лекарств требуют подтверждения inline-кнопкой в Telegram до записи в FHIR.

## 5. Notion: структура

### Пациентская часть

| БД | Поля | Кто пишет |
|---|---|---|
| 🧍 Профиль | ФИО (только имя+отчество, без фамилии если параноишь), DOB, аллергии, группа крови, экстренный контакт | Пациент |
| 📅 Визиты | Дата, врач, клиника, повод, статус, оплата (ОМС/ДМС/платно), ссылки на бриф/SOAP | Агент + пациент |
| 🧪 Анализы | Дата, тип, лаборатория, флаги, ссылка на PDF (локальную) | Агент |
| 📈 Метрики | Дата, показатель (рус+LOINC), значение, норма, источник, флаг | Агент |
| 💊 Лекарства | МНН, торговое, доза, расписание, с какого визита, статус | Агент |
| 🩺 Диагнозы | Состояние, МКБ-10, клиника, статус | Агент |
| 🩻 Исследования | Дата, тип (МРТ/КТ/УЗИ/...), МО, заключение, ссылка на DICOM | Агент |
| 🚨 Алерты | Тип, серьёзность, триггер, дата, статус | Агент |
| 📝 Заметки | Самочувствие, симптомы, вопросы | Пациент |

### Врачебная часть (шаренные страницы)

- **Карточка пациента** — только то, что пациент явно шарит (пермишны в Notion + фильтр в FHIR-слое).
- **Визит-бриф** — 1 страница перед приёмом.
- **Форма заключения** — врач пишет/диктует, `agent.visit_post` подхватывает.

## 6. Приватность, безопасность, юридическая реальность РФ

### Для себя одного

- Диск зашифрован (LUKS/VeraCrypt/FileVault).
- pgcrypto на полях с СНИЛС, полисом, паспортом.
- Notion только под Health OS, 2FA, минимум интеграций.
- Бэкапы — [restic](https://restic.net) + age в локальный NAS или шифрованный [Hetzner Storage Box](https://www.hetzner.com/storage/storage-box) (дёшево, в ЕС, GDPR).
- LLM только локально. Никаких облачных API для медданных.
- [152-ФЗ](http://www.consultant.ru/document/cons_doc_LAW_61801/): обработка своих данных субъектом — вне сферы регулирования как «оператор».

### Как только появляется второй человек (родственник / врач ведёт своих)

Ты становишься **оператором персональных данных специальной категории**. Это означает:
1. **Уведомление в [Роскомнадзор](https://pd.rkn.gov.ru)** — через Госуслуги, бесплатно, но обязательно до начала обработки.
2. **Письменное согласие** на обработку спецкатегории в строгой форме ([ст. 9, 10 152-ФЗ](http://www.consultant.ru/document/cons_doc_LAW_61801/)) — для каждого субъекта.
3. **Оператор обязан обеспечить хранение ПДн россиян на серверах в РФ** ([ст. 18 п. 5 152-ФЗ](http://www.consultant.ru/document/cons_doc_LAW_61801/)). Hetzner (Германия) формально нарушает. Выход: бэкапы на Selectel/Timeweb/VK Cloud, либо локальный NAS.
4. **Политика обработки ПДн** — опубликованная, даже для «на коленке» проекта.
5. **Приказ о назначении ответственного**, регистры, логи доступа.

**Вывод:** MVP **только для себя** и, максимум, второго аккаунта с ручным согласием. Публичный продукт — отдельный проект с юристом, не MVP.

### Врачебная тайна (323-ФЗ ст. 13)

Если врач работает через твою систему — **он** несёт ответственность за разглашение своих пациентов. Твоя роль — **процессор** (обработчик), и между вами нужен договор поручения на обработку ПДн. Для MVP с одним знакомым врачом — письменное соглашение в свободной форме, согласованное юристом, на один лист.

## 7. MVP: две недели

Ты как единственный пациент, один врач (можно ты сам в двух ролях), источник — email от лабораторий + ручной ЕМИАС-экспорт + wearables.

### Неделя 1 — фундамент

**День 1.** Железо и базовая изоляция.
- Mac mini / NUC / старый ноут → Ubuntu 24.04 LTS (или macOS с Docker Desktop).
- Диск зашифрован.
- `docker-compose.yml`: HAPI FHIR, Postgres 16, MinIO, Ollama, Redis (для очередей).
- Локальная сеть: `http://healthos.local` через [Tailscale](https://tailscale.com) или [ZeroTier](https://www.zerotier.com) — чтобы с телефона открывать без выставления наружу.
- SOPS + age для секретов.

**День 2.** Ollama модели.
- Тянем `qwen2.5:14b-instruct-q4_K_M` (если VRAM/RAM позволяет) или `saiga-llama3-8b`.
- Тянем `medgemma-4b-it` для мед.промптов.
- Whisper large-v3.
- Verify: пример промпта про терминологию на русском → адекватный ответ.

**День 3.** FHIR-ядро.
- Patient (я), Practitioner (врач), Organization (моя поликлиника, 2 лабораторий).
- Python-клиент ([`fhir.resources`](https://pypi.org/project/fhir.resources/)).
- Справочник LOINC: минимальный (20 популярных показателей) в Postgres.

**День 4–5.** `ingest.labs.email` v1 на одной лаборатории (возьмём реальное письмо от Инвитро).
- IMAP-пулл, классификация отправителя.
- Docling → текст → LLM-экстракция.
- Запись в HAPI, файл в MinIO.
- Verify: реальное письмо → через 2 минуты в FHIR всё разложено.

**День 6.** Расширение на Гемотест + KDL (если есть образцы писем).

**День 7.** Notion-проекции.
- Создать 9 баз в одном воркспейсе.
- Sync-скрипт: HAPI → Notion upsert по FHIR ID.
- Verify: новый анализ в FHIR → строки в «Анализах» и «Метриках» через 30 секунд.

### Неделя 2 — агенты, цикл визита, алерты

**День 8.** `agent.alerts` v1. Правила по референсам + критические пороги. Telegram-бот через [aiogram](https://aiogram.dev). Создаём бота у [@BotFather](https://t.me/BotFather).

**День 9.** Wearables. Health Auto Export → Cloudflared tunnel → webhook → агрегаты → Observation. Дашборд «Сегодня» в Notion.

**День 10.** `agent.visit_prep`. Google Calendar интеграция. Тег `#visit` в описании события.
- Verify: создаю «Терапевт, 2026-04-27 #visit» → через 30 мин бриф готов.

**День 11.** `agent.visit_post` v1.
- Голосовое в TG → whisper → черновик SOAP в Notion.
- Inline-кнопки: подтверждение каждого изменения лекарств перед записью в FHIR.

**День 12.** Ручной экспорт ЕМИАС.
- Скачиваем свою медкарту с mos.ru (ZIP).
- Парсер CDA → импорт прошлых Encounter, Observation, Condition.

**День 13.** Сквозной прогон.
- Реальный визит к врачу → бриф → (визит) → диктовка → SOAP → новое лекарство → напоминания работают → через 3 дня приходит лаба по почте → обновляются метрики → алерт на отклонение.

**День 14.** Чиним что всплыло, пишем `README.md` для поднятия на новой машине за 30 минут.

### Что НЕ делаем в MVP

- ЕМИАС-автопулл через Playwright (хрупко, делаем на 2-м месяце).
- Публичный веб-UI и мобильное приложение.
- Второй пациент (требует уведомления РКН — отдельный этап).
- ML-предсказания (только rule-based алерты).
- DICOM-просмотрщик (пока ссылка на файл).
- Автоматическая подача на [налоговый вычет за лечение](https://www.nalog.gov.ru) по чекам — соблазнительно, но на v3.

## 8. Что в v2–v3 (через 2–6 месяцев)

- **ЕМИАС Playwright-пулл** — раз в неделю тянет новые осмотры и анализы автоматом.
- **Госуслуги парсинг** — для немосковских регионов.
- **Поддержка DICOM** — встроенный просмотр через [OHIF Viewer](https://ohif.org) локально.
- **Фарма-напоминания** — интеграция с [Еаптека](https://www.eapteka.ru) / [Apteka.ru](https://apteka.ru) для ссылок на покупку и напоминаний о пополнении.
- **Налоговый вычет** — автосбор чеков лечения (письма от клиник + приложения ФНС), формирование декларации 3-НДФЛ.
- **ML baseline** — после 3 месяцев данных по HRV/сну/ЧСС детекция аномалий.
- **Второй пациент** — только после оформления как оператора ПДн (уведомление РКН, согласия, политика).
- **Врачебный web-UI** — минимальный Next.js, когда Notion станет неудобен для нескольких карточек.
