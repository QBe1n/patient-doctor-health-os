# Patient & Doctor Health OS

**Автор:** Богдан
**Дата:** 24 апреля 2026
**Статус:** план MVP, двусторонний (пациент + врач), self-hosted, open-source

---

## 0. Реальность и развилки

Из ответов:

- **Двусторонний** продукт: пациент ведёт свою карту, врач видит шаренный срез, оба получают алерты.
- **Источники:** PDF/фото лабораторных, EHR (Malaffi/Riayati и подобные), ручной ввод + календарь визитов.
- **Сценарии:** подготовка к визиту, авто-обновление карточки, алерты, постобработка визита.
- **Хранение:** всё локально, self-hosted, open-source.

Главный трейдофф, который это задаёт: **Notion перестаёт быть источником истины**. PHI (защищённая медицинская информация) в Notion — нарушение почти любой юрисдикции (HIPAA/GDPR/UAE DHA), и сам Notion в terms запрещает хранить медкарты. Значит:

- **Источник истины — локальный Postgres + HAPI FHIR** (стандарт, с которым уже разговаривают EHR).
- **Notion — это UI/дашборд, куда синкаются агрегаты, заметки, чеклисты, шаренные с врачом срезы.** Сырые PDF/DICOM/лабы не попадают в Notion никогда.
- **Computer (агенты) — оркестратор:** тянет из источников → нормализует в FHIR → пишет в Postgres → обновляет Notion-проекции → шлёт алерты.

Это не «Notion-first проект с AI-обвязкой». Это **локальный Health OS, у которого Notion — одна из поверхностей**.

## 1. Архитектура (high-level)

```
┌─────────────────────────────────────────────────────────────────┐
│                         ПОВЕРХНОСТИ                              │
│                                                                  │
│  Notion (patient)   Notion (doctor)   Telegram bot   Web (opt.) │
│       │                   │               │             │        │
└───────┼───────────────────┼───────────────┼─────────────┼────────┘
        │                   │               │             │
        └───────────────────┴───────────────┴─────────────┘
                            │
                    ┌───────▼────────┐
                    │  Agent Runner  │   ← Computer (cron + event)
                    │   (Python)     │
                    └───────┬────────┘
                            │
   ┌────────────────────────┼────────────────────────┐
   │                        │                        │
┌──▼──────────┐     ┌───────▼────────┐      ┌────────▼─────────┐
│   INGEST    │     │   DOMAIN CORE  │      │    PROJECTIONS   │
│             │     │                │      │                  │
│ • PDF/OCR   │────▶│ HAPI FHIR R5   │─────▶│ Notion databases │
│ • EHR APIs  │     │ (Docker)       │      │ Alerts (push)    │
│ • Wearables │     │                │      │ Visit briefs     │
│ • GCal      │     │ Postgres 16    │      │ PDF заключения   │
│ • Manual    │     │ (encrypted)    │      │                  │
└─────────────┘     └────────────────┘      └──────────────────┘
                            │
                    ┌───────▼────────┐
                    │  Local LLM     │   ← Ollama (Llama 3.1 8B
                    │  + Whisper     │     + medgemma / bio-mistral)
                    └────────────────┘
```

**Почему так, а не «всё в Notion через API»:**

- FHIR — единственный язык, на котором Malaffi, Riayati, Apple Health Records, Epic и пр. умеют говорить. Если с первого дня писать в FHIR, потом не придётся мигрировать.
- Postgres — по факту бэкенд HAPI. Шифрование поверх (LUKS/диск + pgcrypto для чувствительных колонок).
- Ollama + llama.cpp даёт приватный inference. Для суммаризации визитов и извлечения значений из лаб — этого хватает.

### Дефолтный стек (каждый пункт — осознанный выбор)

| Слой | Решение | Почему именно это |
|---|---|---|
| Оркестратор | Python + [Prefect](https://prefect.io) или простой cron | Prefect если > 5 агентов, иначе cron |
| Domain store | [HAPI FHIR](https://hapifhir.io) на Postgres, Docker | Стандарт, не самопал |
| Шифрование | LUKS на диске + pgcrypto для PHI-колонок | Диск зашифрован, ключевые поля — дополнительно |
| Ingest PDF | [Docling](https://github.com/docling-project/docling) + [PaddleOCR](https://github.com/PaddlePaddle/PaddleOCR) | Docling держит структуру, Paddle — фоллбек для фото |
| Нормализация лаб | Regex + LLM → маппинг на [LOINC](https://loinc.org) | 20 базовых показателей хардкодом, остальное LLM |
| LLM | [Ollama](https://ollama.com) → Llama 3.1 8B + [MedGemma](https://deepmind.google/models/gemma/medgemma/) | 8B на M-серии/3090, MedGemma для мед.контекста |
| Speech | [whisper.cpp](https://github.com/ggerganov/whisper.cpp) большая модель | Диктовка визитов |
| Воркеры wearable | [Health Auto Export](https://www.healthexportapp.com) webhook → наш ingest | Apple Health без облака |
| Notion sync | Официальный [Notion API](https://developers.notion.com) | Только проекции, не истина |
| Календарь | Google Calendar API | Уже подключён |
| Алерты | Telegram Bot + email | Telegram у тебя уже в ежедневном стеке |
| Секреты | [SOPS](https://github.com/getsops/sops) + age | Локальные ключи, не облако |
| Деплой | Docker Compose на своей тачке / мини-сервере | Kubernetes не нужен |

Всё это — open-source, всё локально. Никаких managed API для PHI.

## 2. Модель данных (FHIR-first, без фанатизма)

FHIR как язык, но пишем только то, что реально используем. Маппинг для Notion-проекций:

### Core сущности

| FHIR ресурс | Что в нём | Notion-проекция |
|---|---|---|
| `Patient` | ID, ФИО, DOB, пол, контакты | БД «Пациенты» (для врача) / 1 запись (для пациента) |
| `Practitioner` | Врач: ФИО, специализация, клиника | БД «Врачи» |
| `Encounter` | Визит: дата, врач, причина, исход | БД «Визиты» |
| `Observation` | Измерение: показатель, значение, норма, LOINC | БД «Метрики» (фильтр по дате/типу) |
| `DiagnosticReport` | Лаб. отчёт: PDF + набор Observations | БД «Анализы» с attachment-ссылкой |
| `Condition` | Диагноз/состояние, ICD-10 | БД «Диагнозы» |
| `MedicationStatement` | Приём лекарств: препарат, доза, период | БД «Лекарства» + расписание |
| `CarePlan` | План: назначения, цели, следующие визиты | БД «План» |
| `Flag` (наш доп.) | Алерт: тип, серьёзность, триггер, статус | БД «Алерты» |

### Что НЕ хранится в Notion

- Сырые PDF/изображения/DICOM — только локальный object storage ([MinIO](https://min.io) в том же Compose), в Notion — только ссылка на локальный URL (который пациент открывает только с VPN/локально).
- Персональные идентификаторы (паспорт, Emirates ID). В Notion — хэш/псевдоним.
- Полный текст диктовок. В Notion — только суммаризация.

### Пример маппинга одного анализа

```
DiagnosticReport "Биохимия 18.04.2026"
├── Observation "Glucose" (LOINC 2345-7) = 6.2 mmol/L  [норма 3.9–5.5] ⚠
├── Observation "HbA1c" (LOINC 4548-4) = 6.1 %        [норма <5.7]   ⚠
├── Observation "ALT" (LOINC 1742-6)  = 28 U/L        [норма <40]    ✓
└── PDF: /storage/patients/001/labs/2026-04-18-biochem.pdf
```

Notion видит: строку в БД «Анализы» (дата, лаб, флаги), и 3 строки в БД «Метрики» для трендов. PDF не видит.

## 3. Агенты Computer

Computer выступает как **agent runner** — набор задач на расписании + реакция на события. Каждый агент — отдельный Python-модуль с чётким контрактом input → side effects.

### 3.1 `ingest.labs` — разбор лабораторных

**Триггер:** файл упал в `/inbox/labs/` (Dropbox/локальная папка) ИЛИ фото в Telegram-бот.

```
Goal: из PDF/фото получить структурированный DiagnosticReport
Success: запись в FHIR с ≥1 Observation, привязанная к пациенту, PDF в MinIO
Plan:
1. Docling/PaddleOCR → plain text + таблицы
2. LLM extract: {показатель, значение, единица, референс} для каждой строки
3. Маппинг на LOINC (кэш популярных + LLM fallback)
4. Сравнение с референсом → флаги
5. Запись DiagnosticReport + Observations в HAPI
6. Обновить Notion-проекции, поднять флаги в «Алерты»
Verify: в БД «Метрики» появились строки с правильными значениями
Время: 2 дня на v1 (20 показателей хардкодом)
```

### 3.2 `ingest.wearables` — метрики с носимых

**Триггер:** webhook от Health Auto Export (каждый час).

- Принимаем JSON: heart rate, HRV, sleep, SpO2, steps.
- Агрегируем до дневных Observation (лишнее не льём).
- Пишем в HAPI, обновляем дашборд «Сегодня» в Notion.

### 3.3 `ingest.ehr` — EHR-коннекторы (постепенно)

**Реальность про ОАЭ:** Malaffi (Абу-Даби) и Riayati (федеральный) — это HIE между провайдерами. Прямого API «я пациент, дайте мне мою карту» нет ([ClinicSoftware AE обзор](https://www.clinicsoftware.ae)). Варианты:

- **Apple Health Records** в странах, где он работает с клиниками (США — сразу, ОАЭ — частично через DHA-совместимые клиники). Экспорт в JSON → наш ingest.
- **DHA портал** для Дубая — там есть личный кабинет пациента, ручной экспорт PDF → в папку → `ingest.labs`.
- **Прямые запросы** в клинику с просьбой прислать FHIR bundle (имеет смысл только у крупных частных групп).

Вывод: **для MVP забываем про прямой EHR-pull**. Делаем `ingest.labs` + ручной upload через Telegram + wearables. Реальный EHR — v2.

### 3.4 `agent.visit_prep` — подготовка к визиту

**Триггер:** cron каждые 30 мин, проверяет Google Calendar на события с тегом `#visit` в следующие 24 часа.

```
Goal: за 24ч до визита — бриф для пациента и врача
Success: в Notion создана страница «Бриф <дата>» с разделами, ссылка отправлена в Telegram
Plan:
1. Собрать из FHIR: последние лабы, новые Observations с флагами, динамика ключевых метрик за 30/90 дней
2. Собрать открытые Conditions, текущие MedicationStatement
3. LLM: из последних заметок пациента + трендов → 5–7 вопросов врачу
4. LLM: 1-страничный клинический саммари для врача (SOAP-формат)
5. Записать в Notion (пациент видит вопросы, врач — клинический срез)
6. Пуш в Telegram: «Бриф к визиту <врач>, <дата> — посмотреть»
Verify: страница создана, обе стороны получили уведомление
Время: 2 дня
```

### 3.5 `agent.alerts` — алерты по отклонениям

**Триггер:** после каждого `ingest.*` + cron раз в день.

Правила (v1, простые):
- Любой Observation вне референса → Flag severity=low/med/high по дельте.
- Тренд: 3+ дня подряд ухудшения ключевой метрики → Flag medium.
- Пропуск приёма лекарства (нет подтверждения за 6ч от назначенного времени) → Flag low.
- Критические пороги (glucose > 11, SpO2 < 92, HR > 130 в покое) → Flag high + немедленный Telegram.

**Важно:** алерты — не диагноз. Формулировка всегда: «показатель X = Y, вне нормы [A–B], рассмотреть с врачом». Никогда не «у вас диабет».

### 3.6 `agent.visit_post` — постобработка визита

**Триггер:** в Telegram-бот приходит голосовое/текст после визита, либо вручную «запустить постобработку визита <id>».

```
Goal: из сырой диктовки → структурированный Encounter + CarePlan
Success: Encounter с SOAP, новые/изменённые Conditions, MedicationStatements, следующий визит в календаре
Plan:
1. whisper.cpp → текст
2. LLM extract: Subjective / Objective / Assessment / Plan
3. LLM extract: изменения в лекарствах (new/dose change/stop)
4. LLM extract: назначенные исследования, дата следующего визита
5. Запись в HAPI, обновление Notion, создание события в GCal
6. Если назначены анализы — создать задачу «сдать <список>» с напоминанием
Verify: все извлечённые назначения подтверждены пациентом (inline-кнопки в Telegram) до коммита
Время: 3 дня
```

**Human-in-the-loop обязателен.** LLM галлюцинирует дозировки — все изменения лекарств требуют подтверждения кнопкой до записи.

## 4. Notion: структура воркспейса

Две базы (пациент и врач) с разными правами. Для MVP — один воркспейс, просто разные страницы/БД.

### Пациентская часть

| БД | Ключевые поля | Кто пишет |
|---|---|---|
| 🧍 Профиль | 1 запись: ФИО, DOB, аллергии, группа крови, экстренный контакт | Пациент |
| 📅 Визиты | Дата, врач, причина, статус (запланирован/прошёл), ссылка на бриф/SOAP | Агент + пациент |
| 🧪 Анализы | Дата, тип, лаб, флаги (✓/⚠/🔴), ссылка на PDF | Агент |
| 📈 Метрики | Дата, показатель, значение, норма, источник, флаг | Агент |
| 💊 Лекарства | Препарат, доза, расписание, с какого визита, статус (активно/отменено) | Агент (из визитов) |
| 🩺 Диагнозы | Состояние, ICD-10, с какого визита, статус | Агент |
| 🚨 Алерты | Тип, серьёзность, триггер, статус (new/acked/resolved) | Агент |
| 📝 Заметки | Свободные записи пациента (самочувствие, вопросы) | Пациент |

### Врачебная часть (шаренные страницы)

- **Карточка пациента** — свёрнутый SOAP-вид + ссылки на визиты и последние лабы (без сырых PDF, без заметок пациента без его согласия).
- **Визит-бриф** — 1-страничный срез перед приёмом.
- **Заключение** — форма, куда врач диктует/пишет; после сабмита `agent.visit_post` разбирает.

### Шаблоны страниц (главное)

1. **«Сегодня»** — dashboard пациента: открытые алерты, лекарства на сегодня, метрики за 24ч.
2. **«Визит-бриф»** — генерируется агентом: жалобы / динамика / вопросы / лекарства / последние анализы.
3. **«Заключение»** — пост-визит, SOAP + план.

## 5. Приватность и безопасность (на чём нельзя экономить)

- **Диск зашифрован** (LUKS на Linux, FileVault на Mac). Без этого — не начинаем.
- **Postgres:** pgcrypto для полей с именами/ID, TLS для локальных соединений.
- **Notion:** отдельный workspace **только под Health OS**, 2FA, никаких сторонних интеграций. В Notion не попадают: сырые файлы, полные тексты диктовок, паспортные данные.
- **Бэкапы:** ежедневно, шифрованные (restic + age), в Hetzner Storage Box или локальный NAS.
- **Доступ врача:** отдельная Notion-учётка с доступом только к его пациентам. Все шары — через time-limited ссылки, ревью раз в месяц.
- **Логи агентов:** без PHI. Пишем `patient_id`, но не имена и не значения.
- **LLM:** только локальный Ollama. OpenAI/Anthropic API — никогда, даже «для теста».

## 6. MVP: две недели

Single user (ты как пациент), один врач, один источник (ручной upload лаб). Цель — прожить один реальный визит по циклу.

### Неделя 1 — фундамент

**День 1–2:** Docker Compose стек.
- HAPI FHIR + Postgres, MinIO, Ollama (Llama 3.1 8B + Whisper), Prefect/cron.
- `.env` через SOPS. LUKS/FileVault проверен.
- Verify: `curl localhost:8080/fhir/Patient` отвечает, `ollama run llama3.1` отвечает.

**День 3:** Модель. Завести Patient (себя), Practitioner (врача), пустые структуры.
- Python-клиент к HAPI ([`fhir.resources`](https://pypi.org/project/fhir.resources/)).
- Скрипты `create_patient.py`, `create_practitioner.py`.

**День 4–5:** `ingest.labs` v1.
- Docling на один реальный PDF с биохимией.
- LLM extract на 15–20 популярных показателей (хардкод LOINC).
- Запись DiagnosticReport + Observations.
- Verify: кладу PDF в папку — через минуту в FHIR лежит отчёт.

**День 6–7:** Notion-проекция.
- Notion databases по шаблону выше (интеграция, токен).
- Синк FHIR → Notion (только нужные поля, upsert по FHIR ID).
- Verify: новый анализ → новая строка в «Анализы» + записи в «Метрики».

### Неделя 2 — агенты и цикл визита

**День 8:** `agent.alerts` v1. Правила на референсы + 3 критических порога. Telegram bot для пушей.

**День 9–10:** `agent.visit_prep`. Google Calendar → событие с тегом → бриф в Notion + пуш.
- Verify: создаю событие на завтра — через 30 мин бриф в Notion.

**День 11:** `agent.visit_post` v1.
- Telegram-бот принимает голосовое → whisper.cpp → SOAP → черновик в Notion.
- Подтверждение inline-кнопками перед записью лекарств в FHIR.

**День 12:** Wearables.
- Health Auto Export → webhook → Observation дневные агрегаты.
- Дашборд «Сегодня».

**День 13:** Прогон целого цикла.
- Заводим реальный визит в календаре → бриф → (настоящий визит) → диктовка → постобработка → лекарства → алерты на пропуск.

**День 14:** Что не сработало → фиксы. Документация setup (README для себя, чтобы через 3 месяца поднять на новой машине).

### Что НЕ делаем в MVP (сознательно отложено)

- EHR API (Malaffi/Riayati) — нет публичного доступа для пациента, ждём v2 или ручной экспорт.
- Веб-UI — Notion + Telegram достаточно.
- Мультиюзер/многоклиникa — архитектурно готово, но не включаем.
- ML-предсказания — только rule-based алерты, ML после 3 месяцев данных.
- Мобильное приложение — PWA при необходимости, не раньше.
- HL7v2, DICOM viewer — когда появится реальный кейс.

## 7. Что будет в v2 (ориентировочно, месяцы 2–3)

- Реальные EHR-коннекторы (начать с Apple Health Records экспорта, затем DHA портал).

## 8. Открытые вопросы, на которые нужно ответить до начала

1. **Сервер:** крутим на твоём ноуте (работает когда включён) или выделенный мини-сервер (Mac mini / NUC) дома/в офисе? От этого зависит reliability wearable-webhook и cron.
